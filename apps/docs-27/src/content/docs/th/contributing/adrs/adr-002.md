# ADR-002: นามธรรมของฐานข้อมูล

> บันทึกการตัดสินใจทางสถาปัตยกรรมสำหรับรูปแบบการเข้าถึงฐานข้อมูลเชิงวัตถุของ XOOPS

---

## สถานะ

**ยอมรับ** - รูปแบบหลักตั้งแต่ XOOPS 2.0

---

## บริบท

XOOPS ต้องการกลยุทธ์การโต้ตอบกับฐานข้อมูลที่จะ:

1. ไวยากรณ์ SQL เฉพาะฐานข้อมูลที่เป็นนามธรรม
2. ให้การดำเนินการ CRUD ที่สอดคล้องกันในทุกโมดูล
3. เปิดใช้งานการฆ่าเชื้อข้อมูลอัตโนมัติและการหลบหนี
4. รองรับการเปลี่ยนแปลงกลไกฐานข้อมูลในอนาคต
5. ลดความซับซ้อนของการดำเนินการทั่วไปสำหรับนักพัฒนา

ทางเลือกอื่นคือ:
- Raw SQL ตลอดทั้งโค้ดเบส
- เต็ม ORM (หลักคำสอน ฝีปาก)
- นามธรรมน้ำหนักเบาที่กำหนดเอง

## การตัดสินใจ

เราจะใช้ **Handler Pattern** กับ:

### 1. XoopsObject - ที่เก็บข้อมูล

แต่ละเอนทิตีข้อมูลขยาย XoopsObject:
```php
class Item extends XoopsObject
{
    public function __construct()
    {
        $this->initVar('id', XOBJ_DTYPE_INT, null, false);
        $this->initVar('title', XOBJ_DTYPE_TXTBOX, '', true, 255);
        $this->initVar('content', XOBJ_DTYPE_TXTAREA, '', false);
        $this->initVar('status', XOBJ_DTYPE_INT, 0, false);
    }
}
```
### 2. ตัวจัดการ - ผู้จัดการฝ่ายปฏิบัติการ

แต่ละวัตถุมีตัวจัดการที่สอดคล้องกัน:
```php
class ItemHandler extends XoopsPersistableObjectHandler
{
    public function __construct($db)
    {
        parent::__construct($db, 'mymodule_items', Item::class, 'id', 'title');
    }

    // CRUD methods inherited:
    // - create(), get(), insert(), delete()
    // - getObjects(), getCount(), getAll()
}
```
### 3. เกณฑ์ - ตัวสร้างแบบสอบถาม

เงื่อนไขการสืบค้นเชิงวัตถุ:
```php
$criteria = new CriteriaCompo();
$criteria->add(new Criteria('status', 1));
$criteria->add(new Criteria('created', time() - 86400, '>='));
$criteria->setSort('created');
$criteria->setOrder('DESC');
$criteria->setLimit(10);

$items = $handler->getObjects($criteria);
```

## ผลที่ตามมา

### คิดบวก

1. **ความสม่ำเสมอ**: โมดูลทั้งหมดใช้รูปแบบเดียวกัน
2. **ความปลอดภัย**: การหลบหนีอัตโนมัติป้องกันการฉีด SQL
3. **ความเรียบง่าย**: การดำเนินการทั่วไปต้องใช้โค้ดน้อยที่สุด
4. **การบำรุงรักษา**: การเปลี่ยนแปลงเลเยอร์ฐานข้อมูลไม่มีผลกับโมดูล
5. **ความสามารถในการทดสอบ**: สามารถจำลองตัวจัดการเพื่อทำการทดสอบได้

### เชิงลบ

1. **ประสิทธิภาพ**: โอเวอร์เฮดที่เป็นนามธรรมเพิ่มเติม
2. **ความซับซ้อน**: เส้นโค้งการเรียนรู้สำหรับนักพัฒนาใหม่
3. **ข้อจำกัด**: การสืบค้นที่ซับซ้อนอาจต้องใช้ข้อมูลดิบ SQL
4. **ปัญหา N+1**: ไม่มีการโหลดแบบกระตือรือร้นในตัว

### การบรรเทาผลกระทบ

- **ประสิทธิภาพ**: แคชวัตถุที่เข้าถึงบ่อย
- **ข้อความค้นหาที่ซับซ้อน**: อนุญาตข้อมูลดิบ SQL เมื่อจำเป็น
- **N+1**: ใช้ getAll() ด้วยเกณฑ์ที่เหมาะสม

## ตัวอย่างโค้ด

### พื้นฐาน CRUD
```php
$helper = Helper::getInstance();
$handler = $helper->getHandler('Item');

// Create
$item = $handler->create();
$item->setVar('title', 'New Item');
$handler->insert($item);

// Read
$item = $handler->get($id);
$title = $item->getVar('title');

// Update
$item->setVar('title', 'Updated Title');
$handler->insert($item);

// Delete
$handler->delete($item);
```
### แบบสอบถามที่ซับซ้อน
```php
$criteria = new CriteriaCompo();
$criteria->add(new Criteria('status', 'published'));
$criteria->add(new Criteria('category_id', '(1,2,3)', 'IN'));
$criteria->add(new Criteria('created', strtotime('-30 days'), '>='));
$criteria->setSort('views');
$criteria->setOrder('DESC');
$criteria->setLimit(10);
$criteria->setStart(0);

$items = $handler->getObjects($criteria);
$total = $handler->getCount($criteria);
```
